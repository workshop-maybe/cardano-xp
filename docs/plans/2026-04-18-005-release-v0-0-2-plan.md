# chore: ship v0.0.2 release with CHANGELOG, tags, and community attribution

## Overview

Four community-reported issues (#40, #41, #42, #44) shipped today as PRs #45–#48. Issue #43 (public dashboard) is next and will be PR #5 in this batch. Together these five PRs comprise the v0.0.2 release. The repo currently has no release infrastructure — no tags, no GitHub Releases, and its root `CHANGELOG.md` is the Andamio app-template's changelog inherited at fork, describing features we don't own.

This plan establishes release infrastructure, credits the community reporters who made the batch possible, and ships v0.0.2 as a proper artifact.

## Problem Frame

Cardano XP is a public-facing, community-iterative project. When someone files a bug, sees it acknowledged, and sees it shipped — that loop *is* the product. v0.0.2 is the first release that closes the loop with attribution: "You reported this. Here's what we did. You're in the release notes." Without release notes, the loop stays invisible.

Concrete gaps today:

- `package.json` identity (`andamio-app-template@2.0.3-dev-20260307-a`) is template-inherited, not project-specific.
- `CHANGELOG.md` content is the template's. First-time visitors reading it get the wrong story.
- No git tags — no way to reference "the code that shipped as v0.0.X".
- No GitHub Releases — no community-visible announcement surface.

## Requirements Trace

- **R1.** A project-specific `CHANGELOG.md` at repo root describes what cardano-xp has shipped, using [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) format.
- **R2.** The CHANGELOG establishes v0.0.1 as the retroactive baseline (the initial commit of the fork, `2fe19db`, 2026-03-06) and v0.0.2 as the current release covering PRs #45, #46, #47, #48, and the #43 PR (to land).
- **R3.** Every PR in v0.0.2 credits the community reporter by their Andamio alias. Where a reporter has a plausibly identifiable GitHub handle, prefer that; otherwise cite the Andamio alias and the XP-watch report path.
- **R4.** `v0.0.1` and `v0.0.2` exist as annotated git tags pointing at the canonical commits. `v0.0.1` is retroactive on the initial commit.
- **R5.** A GitHub Release is created for `v0.0.2` with the release-notes body copied from the CHANGELOG entry.
- **R6.** Each of the five closed issues (#40, #41, #42, #43, #44) gets a closing comment linking to the v0.0.2 release so reporters see their contribution land.
- **R7.** No Claude Code attribution in commits, PR bodies, CHANGELOG entries, or release notes.

## Scope Boundaries

- Not introducing automated release tooling (`semantic-release`, `changesets`, `release-please`, etc.). The project's cadence is "ship when something's worth shipping"; automation overhead exceeds benefit today.
- Not publishing to npm. Cardano XP is an app, not a library.
- Not creating a release branch. The project ships from `main`.
- Not rewriting the `package.json` name (`andamio-app-template` → `cardano-xp`) in this PR. That's a separate concern — the template name may be load-bearing for `scripts/stamp-version.sh` and build tooling. Flagged as a follow-up decision.

### Deferred to Separate Tasks

- Rename `package.json` from `andamio-app-template` to `cardano-xp`: separate PR after this release lands, with a verification pass on `scripts/` and CI.
- Backfilling v0.0.1 release notes: the baseline tag exists for history, but the initial commit is thin on changelog-worthy detail. Not worth archaeology.

## Key Technical Decisions

- **Source of truth for version: git tags + GitHub Releases.** Not `package.json`. Rationale: the template-inherited `package.json` version string (`2.0.3-dev-20260307-a`) has a dev-timestamp format that `scripts/stamp-version.sh` likely depends on. Changing it risks build breakage. Tags + Releases are orthogonal to package.json and communicate to humans + GitHub tooling without touching code. If a later PR wants to align `package.json`, it can — as its own change with its own verification.
- **Retroactive v0.0.1 tag points at the initial commit (`2fe19db`, 2026-03-06).** This gives v0.0.2 a baseline to range-reference. No release notes for v0.0.1 — the initial commit is effectively "first public state of the fork."
- **CHANGELOG rewrite is full replacement, not append.** The existing file describes the Andamio template's features, not cardano-xp's. Leaving it partially in place would confuse readers. Wipe and start fresh with v0.0.2 at the top and v0.0.1 as the "initial public release" stub.
- **Attribution format: Andamio alias + link to source.** `@username`-style handles aren't reliably verifiable for these reporters — the names come from the Andamio API, not GitHub. Safer to cite as "Reported by `<alias>` via XP feedback" with a link to the issue rather than risk @-mentioning the wrong GitHub account.
- **Phased shipping.** Phase 1 (CHANGELOG + v0.0.1 tag) lands now on its own branch, before #43 starts. Phase 2 (v0.0.2 finalization) lands after #43 merges. Phasing lets #43 plan and review without release pressure, and lets the CHANGELOG infrastructure land on a clean branch.

## Open Questions

### Resolved During Planning

- *Do we credit the `'"name123'' onload='alert()'` reporter literally?* No. Treat as "anonymous contributor (via XP feedback)" and link to the xp-watch report. Literal rendering of an XSS payload in the release notes would defeat the point of #44.
- *Does v0.0.1 need release notes?* No. It's a retroactive baseline tag at the initial commit.
- *Should the CHANGELOG link to individual PRs or to the GitHub Release?* Both. Each entry links to its PR and issue; the top-level version link points at the GitHub Release URL.

### Deferred to Implementation

- Exact commit SHA for the v0.0.2 tag — must wait until #43 merges and we know the final head of `main`.
- Whether `gh release create` supports annotated-tag creation in one step, or whether we `git tag` first and then `gh release create` pointing at the tag. Decide at execution time.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| `scripts/stamp-version.sh` embeds `package.json` version into the build; changing it elsewhere could confuse CI | Don't touch `package.json`. Tags + Releases are the source of truth for this PR. Rename to `cardano-xp` is a documented follow-up. |
| A reporter's Andamio alias is different from their preferred public handle | Cite the Andamio alias + link to the xp-watch report. Don't `@`-mention. If a reporter asks to be credited differently, amend the Release (GitHub allows editing Release bodies). |
| #43 slips or changes scope significantly | Phase 1 (CHANGELOG + v0.0.1 tag) still lands independently. v0.0.2 can be deferred or reduced to 4 PRs with a note in the CHANGELOG. |
| Retroactive `v0.0.1` tag at initial commit is weird | It's a convention widely used (e.g., `kubernetes/kubernetes` has retroactive early tags). Purely a reference anchor. |
